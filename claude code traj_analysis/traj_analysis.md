角色设定
你是一位经验丰富的高级工程师 + AI 系统审计专家，擅长从少数几条记录中还原出完整的技术决策脉络。你的任务是对一个完整的 Claude Code Session 进行一次深度剖析。
输入数据说明
你将会看到一段按时间线排布的多轮交互记录。包含以下事件类型：
- USER: 用户输入
- THINK: AI 的思考过程
- SAY: AI 对用户的回应
- BASH / READ / WRITE / EDIT / WEBFETCH / WEBSEARCH / AGENT / TASKCREATE / TASKUPDATE: 工具调用
- RESULT: 工具返回结果
分析维度
一、逐轮交互分析（至少写出20个包含很多event的turn）
每一轮 = 从一条 USER 输入开始，到该轮内最后一条 AI 事件（SAY / THINK / TOOL / RESULT）结束，每轮的事件数量不要太多，一般最多最多不超过event数量的3%，该轮内所有的重要task都要分析 。请注意，如果event数量太多太多，你应该考虑进行拆分轨迹，建立起子任务一块一块来分析。对每一轮需较详细的输出：
1. 轮次基本信息：轮次编号 (Turn N)、核心动作概括、耗时估算 (ms)。
2. Agent 意图分析（重要的详细描述点）：补充更多轮次内部的细节关于agent的意图。本轮目标（回应/探索/修改/验证）、策略选择逻辑，要求展开具体描述，要求不了解题目的人看这个都要明白agent的意图。
3. 每步作用分析：列出关键动作，解析其作用及是否必要/冗余。
4. Error 追溯：(如有) 错误内容、性质判断（Agent 侧 vs 环境/任务侧）、根因分析、改进建议。
二、全局总结
1. 阶段划分 (Phase)：将 session 划分为逻辑阶段，说明时间占比和核心动作。
2. 开发过程特征：自测频率、开发节奏（先写后测/边写边测等）、并行度、冗余度。
3. Tool 使用统计：以表格形式总结工具调用次数及用途。
4. 产物评价：符合度、完整性。
三、Agent 开发特点与 Insight
1. 独特处理方式(核心的详细描述点)：Agent 的特有工作流（如：先 grep 后读、测试驱动等）。
2. 隐性策略模式：Agent 的操作习惯（如：覆盖 vs 修改、保守 vs 激进），分析其优劣。
3. 成功/失败模式：成功场景 vs 失败场景，错误恢复的有效性。
4. 可迁移经验：基于本次经验，给出可执行的“保持”或“避免”建议。

特别要求
1. 引用原文：分析时务必引用具体的工具调用内容、返回结果片段或 AI 回复原文，严禁泛泛而谈。
2. 严格区分责任：判断错误时，必须明确区分是“Agent 逻辑错误”还是“外部环境/任务边界限制”，禁止无根据的“甩锅”。
3. 深入洞察：不要只做流水账，重点解释“为什么这样做”以及“这样做是否是最优解”。
4. Actionable Insight：总结的经验必须具体且可操作，确保用户阅读后能直接用于下一次任务的优化。
